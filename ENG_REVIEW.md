## Locked Tech Stack

| Layer | Technology | Rationale |
|-------|-----------|-----------|
| Backend API | Laravel (PHP 8.2) + Sanctum + Horizon | Team knows it; ships Day 1; batteries included for queues, auth, testing |
| Database | PostgreSQL + Row Level Security | RLS enforces school_id isolation at DB layer — data leak impossible |
| Queue runner | Laravel Horizon + Redis | Priority queues for AI jobs; native Laravel integration |
| Mobile app | React Native (iOS + Android) | Team knows it; teacher + parent apps from one codebase |
| Web admin dashboard | React + Vite | Team knows React; no SSR needed for internal B2B dashboard; skip Next.js |
| Media storage | AWS S3 (ap-south-1) + CloudFront CDN | India data residency; CDN for parent feed load speed |
| Real-time | Pusher + Laravel Echo | Native Laravel integration; free tier covers pilot phase |
| AI/LLM gateway | OpenRouter (multi-model routing) | 5-10x cost saving vs single-provider lock-in; route by task complexity |
| Face recognition | AWS Rekognition | Only viable server-side face recognition for age 3-6; not replaceable by on-device ML |
| Notifications | Wapi (WhatsApp) → MSG91 (SMS fallback) | Wapi for cost; MSG91 as reliability backstop |
| CI/CD | GitHub Actions | Standard; integrates with existing Cadence Infotech workflow |

## Data Model (Core Tables)

```
MULTI-TENANCY: Every table has school_id.
PostgreSQL RLS policy enforces it on every query.
A query without school_id in context returns 0 rows — not an error.

schools
  id, name, logo_url, brand_color, subdomain
  created_at, updated_at

users  (teachers, parents, admins — unified table, role-based)
  id, school_id*, role (teacher|parent|admin), name
  phone, email, otp_secret, last_login_at
  created_at, updated_at

children
  id, school_id*, name, dob, class_id
  face_embedding_s3_key  (enrollment photo → Rekognition reference)
  dpdp_consent_at        (when parent gave consent for face recognition)
  created_at, updated_at

classes
  id, school_id*, name, academic_year
  teacher_id (FK → users)

parent_child  (many-to-many: one child can have 2 parents)
  parent_id, child_id, relationship (mother|father|guardian)

posts
  id, school_id*, teacher_id, class_id
  media_urls (jsonb — array of S3 keys)
  caption, ai_caption_draft, caption_edited_by_teacher (bool)
  ai_model_used, ai_cost_usd  (for cost tracking)
  status (draft|posted|archived)
  created_at, posted_at

post_children  (which children are tagged in this post)
  post_id, child_id
  confidence_score  (Rekognition confidence — 0-100)
  confirmed_by_teacher (bool)

milestones  (NEP 2020 taxonomy — pre-loaded, ~80 rows)
  id, domain (physical|cognitive|language|social_emotional|aesthetic)
  code, title, description, age_range_months

post_milestones  (AI-suggested + teacher-confirmed milestone tags)
  post_id, milestone_id, suggested_by_ai (bool), confirmed_by_teacher (bool)

engagement_snapshots  (weekly snapshot — never real-time)
  school_id, week_starting, score_pct
  active_parents, total_parents

magic_link_tokens  (for WhatsApp deep links)
  id, parent_id, child_id, token_hash
  expires_at, used_count, max_uses (default 10)
  created_at
```

## Authentication & Authorization Flow

```
TEACHER / ADMIN (mobile + web):
  POST /api/v1/auth/login
    → phone + password
    → returns Sanctum token (30-day TTL)
    → token stored in React Native SecureStore

PARENT (native app):
  POST /api/v1/auth/otp/request  → SMS OTP via MSG91
  POST /api/v1/auth/otp/verify   → returns Sanctum token

PARENT (magic-link web view):
  GET /feed?t={signed_jwt}
    → JWT decoded server-side (24hr TTL)
    → identifies parent_id + child_id
    → web session created (no full login)
    → after 3 views: prompt OTP login
    → OTP login upgrades to full Sanctum token

MIDDLEWARE CHAIN (all authenticated requests):
  1. auth:sanctum          → validate token
  2. SetTenantFromToken    → extract school_id from user, set on DB connection
  3. PostgresRLS           → SET app.current_school_id = {school_id}
                           → RLS policies enforce on every query automatically
```

## AI Pipeline — Caption Drafting

```
Teacher selects photo → taps POST
         │
         v
POST /api/v1/posts  (creates draft post record)
         │
         v
CaptionDraftJob dispatched to 'ai-fast' queue
         │
         v
┌─────────────────────────────────────────────┐
│  CAPTION QUALITY ROUTER                      │
│                                             │
│  Input:                                     │
│    - Photo S3 key                           │
│    - Tagged child names (if confirmed)      │
│    - Class name + school name               │
│    - Time of day                            │
│                                             │
│  Step 1: Gemini Flash (cost: ~$0.0001)      │
│    → Generate caption draft                 │
│    → Score output quality (0-10):           │
│       - Contains child name? +2             │
│       - >10 words? +2                       │
│       - No hallucinated details? +3         │
│       - Warm tone? (sentiment check) +3     │
│                                             │
│  Step 2: Score < 7 → escalate to           │
│    GPT-4o-mini (cost: ~$0.0003)             │
│                                             │
│  Result pushed via Pusher to teacher's      │
│  compose screen                             │
└─────────────────────────────────────────────┘
         │
         v
Teacher reviews draft → edits if needed → posts
         │
         v
Post status → 'posted'
Pusher broadcasts to 'class.{class_id}.feed' channel
Parents receive live update
```

## AI Pipeline — Face Recognition

```
(Runs async AFTER teacher posts — never blocks the post)

Post created → FaceRecognitionJob dispatched to 'ai-slow' queue
         │
         v
For each photo in post:
  → AWS Rekognition: SearchFacesByImage
     against collection: 'school_{school_id}'
     (collection built at child enrollment from reference photo)
         │
         ├── Match confidence > 90%: suggest child tag
         ├── Match confidence 70-90%: suggest with low-confidence flag
         └── Match confidence < 70%: no suggestion
         │
         v
Results saved to post_children (suggested, unconfirmed)
Pusher event → teacher's post view updates with suggestions
Teacher confirms or dismisses each tag
         │
         v
Only confirmed tags visible to parents

DPDP GATE: Rekognition only called if child's dpdp_consent_at IS NOT NULL
If no consent: post goes through, face tags skipped entirely
```

## Magic-Link Flow (Parent Onboarding)

```
School admin adds parent phone → system generates magic_link_tokens record
         │
         v
Wapi sends WhatsApp message:
  "[School name]: Rohan had a great morning!
   See his update → https://cs.app/f?t={token}"
         │
         v
Parent taps link → browser opens /f?t={token}
         │
         v
Server validates token (not expired, not > 10 uses)
  → Extracts parent_id, child_id from token
  → Creates guest session (not full auth)
  → Increments used_count
         │
         v
React web view renders:
  → Feed-first (full-bleed photo + caption)
  → No login prompt on first 3 views
         │
         v
After 3 views (used_count >= 3):
  → Banner: "Get the app for daily updates"
  → OTP prompt on next action
         │
         v
OTP login → upgrades to full Sanctum session
         │
         v
DPDP consent screen shown (first time only)
  → Per-type checkboxes (3 items)
  → Stored in children.dpdp_consent_at + consent_items (jsonb)
```

```

===================================
auth/
  ├── [REQUIRED] Teacher login → valid token
  ├── [REQUIRED] Teacher login → invalid password → 401
  ├── [REQUIRED] Parent OTP request → SMS sent
  ├── [REQUIRED] Parent OTP verify → valid code → token

  ├── [REQUIRED] Magic link → valid token → guest session
  ├── [REQUIRED] Magic link → expired token → 410
  └── [REQUIRED] Magic link → used_count > 10 → 410

MULTI-TENANCY (CRITICAL — data leak risk)
===================================

  ├── [REQUIRED E2E] School A teacher CANNOT read School B posts
  ├── [REQUIRED E2E] School A parent CANNOT see School B children
  ├── [REQUIRED] RLS enforced even if school_id omitted from query
  └── [REQUIRED] Admin of School A CANNOT access School B admin API

DPDP CONSENT
===================================
dpdp/
  ├── [REQUIRED] Rekognition NOT called if dpdp_consent_at IS NULL
  ├── [REQUIRED] Consent revocation → Rekognition data deleted in job
  ├── [REQUIRED] Consent screen shown ONCE per parent per child
  └── [REQUIRED] Consent audit log written on every change

POST CREATION FLOW
===================================
posts/
  ├── [REQUIRED] Post creates → caption job dispatched
  ├── [REQUIRED] Post creates → face recognition job dispatched
  ├── [REQUIRED] Caption job timeout → post not blocked (empty caption OK)
  ├── [REQUIRED] Face recognition fails → post not blocked (no tags OK)
  ├── [REQUIRED] Video > 10MB client-side → rejection before upload
  ├── [REQUIRED E2E] Teacher posts → parents receive Pusher event
  └── [REQUIRED] Pre-signed URL → upload → post status updated
```

```

  ├── Cursor pagination returns correct page

AI JOBS
```

## Cost Model (Unit Economics)

This section exists so you can validate pricing **after** knowing costs.

### Per-school per month estimates (50 children, 5 classes, 3 teachers)

| Cost item | Usage assumption | Monthly cost (USD) |
|-----------|-----------------|-------------------|
| **S3 media storage** | 3 posts/class/day × 5 classes × 30 days × 2MB avg | ~$0.09 |
| **CloudFront CDN** | 50 parents × 3 posts/day × 2MB | ~$0.45 |
| **Rekognition** | 150 photos/day × 30 days × $0.001/image | ~$4.50 |
| **OpenRouter (captions)** | 150 posts/day × 30 days × $0.0002 avg | ~$0.90 |
| **Pusher** | 150 events/day × 50 connections × 30 days | Free tier |
| **MSG91 SMS** | 50 parents × 30 days × 1 notif × ₹0.15 | ~₹225 (~$2.70) |
| **Laravel Horizon/Redis** | Shared infra — amortize across schools | ~$0.50 |
| **PostgreSQL** | Shared infra — amortize across schools | ~$0.30 |
| **TOTAL (per school/month)** | | **~$9.50 (~₹790)** |

**At ₹20/child/month × 50 children = ₹1,000 revenue per school.**
**Cost: ~₹790. Gross margin: ~₹210 (~21%).**

This is very thin at launch. Key lever: **Rekognition at $4.50/school/month is 47% of your cost.**

### Cost optimization levers (in priority order):

1. **Rekognition cost**: Only run face recognition on photos (not videos). Batch process at end of day rather than per-post. At scale, train on-server model to replace Rekognition (Phase 2). → Saves ~$3/school/month.
2. **Caption cost**: On-server fine-tuned small model for simple captions (Phase 2) → saves ~$0.80/school/month.
3. **Storage**: Tiered retention already decided → saves ~80% of storage cost after Year 1.
4. **SMS fallback**: Only send SMS when WhatsApp delivery confirmed failed (not as default) → saves ~$1.50/school/month.

**After Phase 2 AI optimization: cost drops to ~$3-4/school/month → margin improves to ~₹600-700 (~60-70%).**

**Pricing recommendation (from cost model):** Don't fix ₹20-30/child/month as permanent. Pilot at ₹20/child. Raise to ₹30/child at Month 3 when you have NPS data. Raise to ₹50/child at Month 12 once AI costs are optimized. The cost model supports ₹50/child at 70% margin.

## New TODOS from Engineering Review

Add these to TODOS.md:

- [ ] **RLS integration test** — Write a test that proves School A's teacher cannot read School B's data. This is the most critical test in the codebase. A missing `school_id` filter in any query must return 0 rows, not leak data. Write before any feature code.

- [ ] **DPDP gate test for Rekognition** — Write a test that proves `FaceRecognitionJob` skips the Rekognition API call when `children.dpdp_consent_at IS NULL`. Must run in CI. DPDP violation if this fails silently.

- [ ] **Rekognition collection init on school signup** — When a school is created, automatically create its Rekognition face collection (`school_{id}`). If this is missing, face recognition jobs crash silently. Add to school onboarding flow (Month 1-2 backend work).

- [ ] **S3 upload retry UI** — If the direct S3 upload fails (network drop mid-upload), the post gets stuck in `draft` status with no UI feedback. Add: client-side retry with exponential backoff + "Upload failed — tap to retry" toast. Teachers on Indian mobile networks will hit this.

- [ ] **Pusher offline fallback** — If Pusher is unavailable, the parent feed doesn't update until next app open. Add: on Pusher reconnect, fetch posts since last_seen_at. This prevents parents from missing posts during Pusher downtime (~5% of time on Pusher free tier).
